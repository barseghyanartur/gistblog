A rule of thumb

Always code as if the dev who ends up maintaining your code will be a violent psychopath who knows where you live.

Introduction

Documentation of every software project should minimally provide (preferably - quick) answers to the following questions:

  • What it is and what it does (overview)?
  • How to install?
  • How to configure?
  • How to run?
  • How to use?
  • How to test?
  • How to troubleshoot?
  • How to contribute?

Reusable software adds more questions to answer:

  • What are common use cases?
  • What are advanced use cases?
  • How to customize?
  • How to migrate?
  • Is documentation versioned?

Open-source adds even more:

  • How and where to ask questions?
  • Security policy?
  • Deprecation policy?
  • License?
  • How to request features?
  • How to report bugs?

How to start

Let's go though a template.

Overview

Briefly describes the general context of the project and several of its aspects, such as the domain, goals, etc. Often includes links to more in-depth view of the project (architecture).

Example:

# Awesome Z

The awesome project for showing off awesomeness.
Consists of a frontend (ReactJS), backend (FastAPI) and serverless ML.
Used primarily to bootstrap software development of Slimmer AI in AWS cloud.
See the [architecture.md](docs/architecture.md) for architectural details.

Prerequisites

List the system and software prerequisites that are required to use the project. Provide clear instructions for how to install any necessary packages or dependencies.

Example:

## Prerequisites

### System packages

You will need `System.Package.Name.A` and `System.Package.Name.B`:

#### Linux

```sh
sudo yum install system-package.name.a system-package-name.b
```

#### Mac

```sh
brew install system-package.name.a system-package-name.b
```

#### Windows

```sh
Install-Package -Name system-package.name.a ...
Install-Package -Name system-package.name.b ...
```

### Infrastructure

You will need:

- AWS account with Admin rights for running CDK
- AWS Access Key and Secret Key
- AWS S3 cloud bucket

See all the details in [Infrastructure prerequisites](docs/infra_prerequisites.md).

Usage

Describe how to use the project. Provide examples and use cases to help users understand how to apply the information.

## Usage

1. Run ``make deploy {env}``

3. ...

Contributor Guidelines

Provide guidelines for how to contribute to the project, including coding conventions and standards, version control practices, and testing and debugging procedures. This is often a separate document.

Example:

# Contributor guidelines

## Developer prerequisites

### pre-commit

Refer to pre-commit for installation instructions.

TL;DR:

```sh
pip install pipx --user  # Install pipx
pipx install pre-commit  # Install pre-commit
pre-commit install  # Install pre-commit hooks
```

Installing pre-commit will ensure you adhere to the project code quality standards.

### Code standards

black, isort, ruff and doc8 will be automatically triggered by pre-commit.

### Requirements

Requirements are compiled using pip-tools.

```sh
make pip-compile
```

### Virtual environment

You are advised to work in virtual environment.

TL;DR:

```sh
python -m venv env
pip install -e .
pip install -r requirements/dev.txt
```

### Documentation

Explain, how to write documentation.

- The documentation uses [MkDocs](https://www.mkdocs.org/).
- All the documentation should be written in Markdown format and reside (except for the ``README.md``) in the ``./docs/`` directory.

### Testing

See [testing](README.md#testing) for more.

### Pull requests

You can contribute to the project by making a pull request.

For example:

- To fix documentation typos.
- To improve documentation (for instance, to add new recipe or fix an existing recipe that doesn’t seem to work).
- To introduce a new feature (for instance, add support for a non-supported file type).

**Good to know:**

Test suite makes extensive use of parametrization. Make sure you have added your changes in the right place.

**General list to go through:**

- Does your change require documentation update?
- Does your change require update to tests?
- Does your change rely on third-party cloud based service? If so, please make sure it’s added to tests that should be retried a couple of times. Example: ``@pytest.mark.flaky(reruns=5)``.

**When fixing bugs (in addition to the general list):**

- Make sure to add regression tests.

**When adding a new feature (in addition to the general list):**

- Check the licenses of added dependencies carefully and make sure to list them in prerequisites.
* Make sure to update the documentation (check whether the installation, features, recipes and quick start require changes).

### Questions

Explain, how (and where) questions could be asked.

### Issues

- Explain how to fill a regular bug.
- Explain how to fill a security bug.

Documentation

Include links to any relevant documentation or resources, such as API references or user guides.

Example:

## Documentation

Searchable documentation is hosted on [GitHub](http://github.io/your-company/your-project). Builds are triggered automatically.

- For tips on writing documentation, see the [contributor_guidelines.md](docs/contributor_guidelines.md).
- Check [recipes](docs/recipes.md) for simple and complex examples.

Recipes

A common ways of performing tasks that framework/package aims to cover. This is often a separate document.

## Recipes

### How to add a new serverless API

Briefly describe the use case. For full code see `sample_app_1 <examples/sample_app_1/>`__.

- Run ``make add_api_module``
- Do you need database models? If so, define them in ``models.py``
- Register new endpoints in ``routes.py``
- ...

#### Database models

**sample_app_1/models.py**

```python
from sqlmodel import Field, SQLModel

class Profile(SQLModel, table=True):
    id: int = Field(default=None, primary_key=True)
    name: str
    secret_name: str
    age: int = None
```

#### API views

```python
import json

def hello(event, context):
    # ...
    return {"statusCode": 200, "body": json.dumps({"message": "hello unknown!"})}

def lambda_handler(event, context):
    return hello()
```

#### Routes

**sample_app_1/routes.py**

```python
from aws_lambda_powertools import ...

app = APIGatewayRestResolver()

def lambda_handler(event: dict, context: LambdaContext) -> dict:
    return app.resolve(event, context)
```

Recommendations

Based on best practices for effective technical developer documentation, consider the following:

Understand users' needs

Tailor documentation to users' level of expertise and provide context and use cases to help users understand how to apply information.

Break content into chunks

Break content into small, manageable chunks that are easy to digest.

Use multimedia elements

Use multimedia elements to supplement text-based documentation and provide visual aids.

Consistent formatting and style

Use consistent formatting and style throughout the documentation to make it easy to navigate and understand.

Encourage user feedback and involvement

Encourage user feedback and involvement in the documentation process to improve its accuracy and relevance.

Support

Provide information about how to get help or support, including contact information and any relevant troubleshooting or FAQ resources.
